5.   Driver Usage

 

Unzip the zipped distribution file.  This will result in a single top-level folder.  Place this folder in <LabVIEW Installation Directory>\instr.lib (if LabVIEW was installed in the default location, <LabVIEW Installation Directory> will be something like ‘C:\Program Files\National Instruments\Labview x.x’, where x.x refers to the LabVIEW version number.  Then restart LabVIEW, which will automatically generate the palettes). 

 

This driver makes heavy use of typedefs – almost every control / indicator on the VIs has a corresponding typedef.  Typedefs are used because they are useful for parameter coercion (i.e. allowing the user to only enter values within a given parameter’s range and minimum increment/granularity size), and in addition provide some level of type safety.  To take advantage of the large number of typedefs, the suggested programming approach is to find the VI you need in the KE622x palette, place it on the block diagram, then right-click on the inputs to create the controls via the pop-up menu (this makes it trivial to add controls since you don’t have to search through a directory to find the typedef you are looking for).  Finally, note that the range and increment coercion provided via the typedefs only works if the inputs to a given VI are driven directly from a front panel control.  The implication here is that programmatically calculated values passed in to VIs will not be coerced by LabVIEW (e.g., suppose that the current level passed into a given VIs was not entered directly from a front panel control, but was instead a calculated value based on some other control’s setting).  For values that are out-of-range, the instrument will report an error.  However, the instrument will not report errors for values that violate the minimum increment size, but will instead silently coerce the value for you.  Every VI has one optional control to help deal with these situations, ‘Check for SCPI Errors?’  If ‘Check for SCPI Errors?’ is True, an error query is sent to the instrument after the command is sent.  In addition, many VIs, particularly those that set analog values (and hence a have a continuous range), have a ‘Read Back Set Value(s)?’ control.  If this control is set to True, the instrument will be queried for the actual set value, which can then be checked to see if the instrument performed any coercion (the set value is made available to the user via an output parameter of the specific VI).  Note that both controls default to True for ‘bullet-proofing’.  Obviously, querying the instrument for errors and coercions comes with a speed penalty.  So, it is recommended that user-created VIs have hooks to set these switch values to True or False.  During initial program debug, it is best to leave these at their defaults (True).  Once the user is satisfied that the VI is operating correctly, they can be set to False to speed-up program operation. 

 

Finally, it is suggested that you only use the VIs that are in the palette set, at least when you are starting out.  These are typically high-level VIs that set a number of parameters in the instrument via a single VI and an input cluster control.  Refer to the ‘Examples’ palette to see how these VIs are used.
